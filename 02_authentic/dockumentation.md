# Демонстрационный пример: OAuth 2.0 и аутентификация пользователя

Данный пример показывает минимальное приложение, в котором **клиентское приложение** получает доступ к защищённому API от имени пользователя с использованием **OAuth 2.0** и (как правило) **OpenID Connect (OIDC)** для подтверждения личности пользователя.

> **Важно для учебных целей.** В стандарте [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749) OAuth 2.0 описывает **делегирование авторизации** (выдачу токенов доступа). Сами по себе «войти по OAuth» без дополнительного слоя не гарантируют единого формата данных о пользователе. **Аутентификация** (кто пользователь) в экосистеме веб-приложений чаще всего реализуется через **OIDC** — профиль поверх OAuth 2.0, который добавляет **ID Token** и стандартизированные claims об identity. В учебном тексте уместно говорить: «OAuth 2.0 + OIDC» или «OAuth 2.0 для доступа к API, OIDC — для входа».

## Учебные роли (терминология OAuth 2.0)

| Роль | В этом примере (типичное соответствие) |
|------|----------------------------------------|
| **Владелец ресурса (Resource Owner)** | Пользователь |
| **Клиент (Client)** | Frontend (публичный или конфиденциальный — зависит от типа приложения) |
| **Сервер авторизации (Authorization Server)** | Authentic — выдаёт токены после аутентификации и согласия |
| **Сервер ресурсов (Resource Server)** | Backend — принимает запросы с **access token** и проверяет его |

## Структура приложения

Приложение состоит из следующих компонентов:

- **Authentic** — сервер идентификации и авторизации (Identity Provider / Authorization Server): хранит учётные записи, выдаёт токены.
- **Frontend** — клиентское приложение: перенаправляет пользователя на вход, получает результат авторизации (например, код авторизации или токены — в зависимости от выбранного потока), обращается к Backend.
- **Backend** — защищённый API (Resource Server): проверяет **Bearer**-токен в заголовке `Authorization` (или другой согласованный механизм) и возвращает данные только для валидного запроса.

Запуск — в Docker с **Docker Compose**. Конфигурация описана декларативно («инфраструктура как код») и поднимается при первом запуске.

## Authentic

В репозитории роль **Authentic** выполняет **Keycloak** (образ `quay.io/keycloak/keycloak:26.0`). При первом запуске realm **`demo`** импортируется из файла `keycloak/import/demo-realm.json` (при необходимости его можно пересобрать скриптом `python3 scripts/generate_realm.py`).

Консоль администратора Keycloak (не путать с пользователями realm): **http://localhost:8080/** — учётная запись bootstrap задаётся в `docker-compose.yml` (по умолчанию `admin` / `admin`).

По умолчанию в realm `demo` создаётся тестовая среда:

- пользователь **admin** / пароль **admin** — роль realm `admin`;
- роль **user**: пользователи `user1` … `user10`, пароли `password1` … `password10`;
- роль **power_user**: отдельные логины `puser1` … `puser10`, пароли `password1` … `password10` (логины уникальны, чтобы не конфликтовать с `user*`).

OIDC-клиент для Next.js: **`demo-frontend`** (публичный клиент, **Authorization Code + PKCE**), redirect: `http://localhost:3000/*`.

Рекомендуется **сменить пароли** при первом входе в продакшен-подобной среде (в учебном стенде это может быть отключено ради простоты).

### Запуск стека

Из корня репозитория:

```bash
docker compose up --build
```

(или `docker-compose`, если установлен старый плагин CLI.)

Сервисы: Keycloak **8080**, backend (FastAPI) **8000**, frontend (Next.js) **3000**. После входа на **http://localhost:3000** нажмите «Войти», затем «GET /me на backend» — запрос уйдёт с заголовком `Authorization: Bearer <access_token>`.

## Frontend

Простое приложение на **Next.js**: после успешного входа через Authentic отображает идентификатор пользователя (например, логин или `sub` из OIDC). Затем запрашивает данные у Backend, передавая **доказательство авторизации** — как правило **access token** (иногда вместе с метаданными сессии), в соответствии с выбранным потоком OAuth 2.0.

## Backend

Приложение на **Python (FastAPI)** с одним или несколькими защищёнными маршрутами: извлекает токен из запроса, **валидирует** его (подпись, срок действия, audience/issuer — в зависимости от настройки) и возвращает ответ с информацией о текущем пользователе или ресурсе.

### Как работает JWKS Client (проверка подписи access token)

**Access token** в OIDC обычно оформлен как **JWT** (три части, разделённые точкой: заголовок, полезная нагрузка, подпись). Подпись создаётся **асимметричным ключом** сервера авторизации (например, RSA). Чтобы **не хранить** на Resource Server статический публичный ключ и **не угадывать**, каким ключом подписан токен, используется стандарт **JWK** ([RFC 7517](https://datatracker.ietf.org/doc/html/rfc7517)) и набор ключей **JWKS** (JSON Web Key Set).

Поток проверки в этом проекте:

1. **Эндпоинт JWKS.** У OpenID Provider зафиксирован URL набора ключей. Для Keycloak в realm `demo` это  
   `{issuer}/protocol/openid-connect/certs`  
   (в терминах OIDC — это `jwks_uri` из [Discovery](https://openid.net/specs/openid-connect-discovery-1_0.html)). В `docker-compose.yml` для backend задано **`JWKS_URL`**: из контейнера backend запрос идёт на сервис Docker **`http://keycloak:8080/.../certs`**, потому что по имени `keycloak` резолвится контейнер IdP; это не обязано совпадать с **issuer** в самом JWT (у пользователя в токене обычно `iss: http://localhost:8080/realms/demo`).

2. **`PyJWKClient` (библиотека PyJWT).** При инициализации клиент знает только **`JWKS_URL`**. Он **HTTP GET** запрашивает JSON с массивом ключей `keys[]` (каждый элемент — JWK: `kty`, `kid`, `n`, `e` для RSA и т.д.). Клиент **кэширует** ответ (с возможностью обновления по TTL и при ошибках подписи — поведение зависит от версии PyJWT; в учебном коде важно сам факт получения ключей с IdP).

3. **Выбор ключа по `kid`.** В заголовке JWT есть поле **`kid`** (key id). `PyJWKClient.get_signing_key_from_jwt(token)` извлекает `kid` из токена, находит соответствующий JWK в загруженном JWKS и возвращает **публичный ключ** для проверки подписи. Если `kid` не найден или JWKS недоступен, проверка не проходит.

4. **`jwt.decode`.** После получения ключа вызывается `jwt.decode(..., algorithms=["RS256"], issuer=OIDC_ISSUER, ...)`. Проверяются **подпись**, срок **`exp`**, **`iss`** (должен совпадать с **`OIDC_ISSUER`** — в нашем случае `http://localhost:8080/realms/demo`, как в токене у пользователя). Проверка **`aud`** отключена (`verify_aud: false`), потому что у Keycloak в access token поле audience часто не совпадает с client id; вместо этого проверяется claim **`azp`** (authorized party) — токен должен быть выдан для клиента **`demo-frontend`** (`OIDC_CLIENT_ID`).

Итог: **JWKS Client** — это слой между Resource Server и IdP: он подтягивает актуальные **публичные** ключи с сервера авторизации и выбирает нужный по **`kid`**, после чего PyJWT проверяет, что JWT не подделан и выдан доверенным **issuer**.

## Какой поток OAuth 2.0 обычно выбирают

- **Веб-приложение с backend** — часто **Authorization Code** с **PKCE** ([RFC 7636](https://datatracker.ietf.org/doc/html/rfc7636)), особенно для публичных клиентов.
- **Классический серверный веб-клиент** — Authorization Code с секретом клиента на сервере Next.js/API routes (модель зависит от архитектуры).

В этом репозитории на frontend используется **grant type** `authorization_code` с **PKCE** (S256), **response_type** `code` (через библиотеку Auth.js / NextAuth и провайдер Keycloak). В описании лабораторной работы полезно приложить диаграмму последовательности (Resource Owner → Client → Authorization Server → Client → Resource Server).

## Краткий итог для студентов

1. **OAuth 2.0** — про выдачу **access token** для доступа к API.  
2. **OIDC** — про **аутентификацию** и стандартные claims о пользователе (**ID Token**).  
3. **Backend** не должен доверять «логину в теле запроса» от клиента без проверки токена (или без доверенного канала server-to-server, если так задумано архитектурно).  
4. Секреты клиента и ключи хранятся только на стороне, где им доверяют (не в публичном фронтенде без мер защиты).
